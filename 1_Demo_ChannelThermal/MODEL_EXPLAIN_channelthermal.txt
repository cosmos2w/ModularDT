Demo 1 Channel Thermal Forward Model
====================================

Updated: 2026-04-28

Stage A: local module surrogate
-------------------------------
The local surrogate learns one circular heated module in normalized local
coordinates. Inputs are known-before-solve module parameters and port condition
tokens:

  module_params = [q_internal, solid_k, solid_alpha, h_mean, h_std,
                   T_env_mean, T_env_std]
  port_tokens = [theta, cos_theta, sin_theta, T_env, h]

The port tokens are boundary and condition inputs only. They do not include
solved surface temperature or solved heat flux.

The local target generator uses the same raw thermal scale as the global
module interiors by solving with an effective normalized-disk coefficient
solid_alpha / module_radius^2. The default local T_env and h sampling ranges
cover the global interface_condition T_outside and h_proxy ranges.

For the current Stage-B workflow, train this local surrogate with
normalize_inputs=true and normalize_targets=true. The checkpoint stores local
normalizer stats. Stage B applies those stats when calling the frozen local
surrogate and rescales the local outputs to the global target convention used
by the Stage-B loss.

The model encodes module parameters, encodes all port tokens, aggregates the
boundary tokens with a small Perceiver-style cross-attention stack, and produces
a compact module_response_latent. Two decoders are supervised:

  internal_temperature(xi, eta)
  interface_pred(theta) = [T_surface, q_normal]

The module_response_latent is saved by the forward pass so the Stage B global
model can use the learned local thermal response as a module token.


Stage B: global channel thermal model
-------------------------------------
The global model predicts the steady channel field in nonperiodic coordinates:

  [u, v, p, omega, temperature]

It uses the global packed HDF5 case structure:

  Re, U_in, material parameters
  module_centers, heat_powers, module_present
  sampled query points and field targets
  module internal temperature targets
  interface_condition and interface_target

The global simulator that produces these targets is lightweight and
approximate. The flow is analytic/diagnostic, not a full Navier-Stokes solve.
It records flow diagnostics and can optionally run a simple projection cleanup,
but projection is disabled by default.

Temperature is advanced dynamically with explicit substeps until convergence or
the configured maximum solve time. The preprocessor now packs steady targets
from converged frames by default instead of blindly averaging the final frames
of a fixed-duration transient.

Global target modes:

  converged_final: use the first saved frame marked converged; recommended for
                   first steady forward-model training
  converged_window_mean: average a final window ending at the converged frame
  final_window_legacy: reproduce the old final heat-active window behavior

Inspect convergence in raw case frame_index.csv, visualization output
convergence_temperature.png, and processed preprocessing_case_quality.csv.

The geometry is nonperiodic. Relative features use direct displacement:

  dx = query_x - module_x
  dy = query_y - module_y
  distance, downstream, upstream
  bottom/top wall distances
  inlet/outlet distances

No minimum-image or periodic wrapping is used.


Environment tokens
------------------
The model creates a fixed nonperiodic grid of environment tokens. Each token
contains normalized x/y position, wall distance, inlet distance, outlet
distance, and a simple channel-centerline encoding. These tokens are combined
with the global operating token.


Hypergraph organizer
--------------------
The organizer builds soft assignment matrices:

  A_me: module-to-environment context
  A_mh: module-to-hyperedge assignment
  A_eh: environment-to-hyperedge assignment

In this thermal channel demo, a hyperedge means a learned thermal/flow
interaction group. It may represent a heat plume, a wall-influenced zone, or a
broader module/environment coupling region. It is not limited to vortex or wake
semantics.

Physically, one hyperedge should combine:

  source-side module support
  channel/environment region support

Examples include modules sharing a downstream thermal plume, near-wall modules
and their adjacent wall/channel region, an upstream module group affecting
downstream coolant temperature, or a thermal cluster with a common local
interface response.

The module-to-environment map A_me is still learned by attention plus geometry
bias, but Stage B now weakly supervises it with a nonperiodic channel prior:
near tokens, laterally aligned downstream tokens, wall-adjacent tokens for
near-wall modules, and heat-power magnitude all receive interpretable mass.
This prior is not a hard rule; it only keeps the organizer from becoming an
unlabeled latent attention block.

The most important direct organizer loss supervises A_eh with a factor target:

  A_eh_target[e,k] proportional to sum_i A_me_for_target[i,e] * A_mh[i,k]

By default A_me_for_target is the thermal prior. This says an environment token
should choose hyperedge k when the modules influencing that token also belong
to k. It directly couples environment-region ownership to module grouping.

The model also aligns normalized hyperedge masses:

  module mass from A_mh over valid modules
  environment mass from A_eh over environment tokens

The alignment uses smooth L1 in log space by default. It prevents one-sided
hyperedges that own modules but no channel region, or a channel region but no
source modules.

Hyperedge strength is no longer a standalone sigmoid head. The semantic
diagnostic is:

  hyper_strength = sqrt(hyper_module_mass * hyper_env_mass)

so strength is high only when both source modules and environment support are
present. The old learned sigmoid is retained only as hyper_strength_learned.

A_eh scoring can also include hyperedge-specific nonperiodic relative geometry
from each provisional A_mh-weighted source coordinate to each environment
token. This helps different hyperedges claim different physical channel
regions.

DISABLE_EDGE remains available for optional visualization experiments, but it
is not recommended as a default for this first steady model. Early hard pruning
can hide whether the weak organizer losses are fixing the learned organization.

The organizer exposes hyperedge state, source coordinates, thermal-region
coordinates, module/environment masses, semantic strength, and the assignment
matrices for diagnostics and visualization.


Port condition head
-------------------
The global model predicts per-module local-surrogate port condition tokens.
The first three token columns are known geometry:

  theta, cos_theta, sin_theta

The head predicts only physical condition values:

  T_env, h

Training supervises these values against the global dataset's clean
interface_condition columns T_outside and h_proxy.


Local surrogate integration
---------------------------
When configured with use_local_surrogate=true, train.py loads a Stage A
LocalModuleSurrogate checkpoint. If freeze_local_surrogate=true, the local
model is held fixed.

The global model can call the local surrogate with:

  local_module_params
  teacher-forced, predicted, or mixed port tokens
  local internal query points

The returned internal temperature, interface prediction, and
module_response_latent are inserted into the global model. The first training
default is teacher-forced port conditions so Stage B can learn a stable global
organizer before relying on its own port predictions.

The recommended curriculum is teacher -> mixed -> predicted:

  teacher_epochs: use dataset-derived local port tokens
  mixed interval: linearly blend teacher and predicted port tokens
  predicted after the curriculum: use only model-predicted port tokens

This matches the evaluation distinction. Teacher mode is a debugging aid
because it supplies exact dataset port conditions to the local surrogate.
Predicted mode is the realistic forward-design mode because, during autonomous
inference, no solved teacher port tokens are available.


Global neural-field decoder
---------------------------
The decoder receives query_xy, global operating context, organized module
tokens, environment tokens, hyperedge tokens, and nearby nonperiodic module
geometry. It predicts the steady field at arbitrary point queries.


Losses
------
Stage A losses:

  internal temperature MSE
  interface target MSE
  optional interface smoothness penalty

Stage B losses:

  global field MSE over [u, v, p, omega, temperature]
  internal module temperature MSE over valid module disk points
  interface MSE over [T_surface, q_normal]
  port condition MSE over [T_env, h]
  weak A_me KL to the channel thermal module-environment prior
  A_eh factor KL from A_me_for_target and A_mh
  module/environment hyperedge mass alignment
  diag_organizer_strength_mean for organizer monitoring

diag_organizer_strength_mean is logged only. It is not added to the total loss;
a positive organizer_strength_weight historically penalized strong hyperedges
and should usually remain 0.0.

This ChannelThermal model intentionally does not copy the dynamic/periodic
machinery from 0_Demo_MultiCylinder: no phase input, no dynamic residual branch,
no periodic wake prior, no frequency loss, and no duplicate-edge pruning loss by
default. The problem here is a steady nonperiodic channel thermal field, so the
organizer supervision stays direct and lightweight.


Global samples and local targets
--------------------------------
The Stage-B packed dataset stores global sampled_points for the fluid/channel
field and separate module_internal_temperature targets for solid interiors. The
global preprocessor now excludes module-interior cells from sampled_points by
default, so the global field loss and local solid-temperature loss supervise
different physical regions. The compatibility flag
--no-exclude-module-interior-from-global-points restores the old behavior.


Evaluation modes
----------------
evaluate.py supports:

  --local-port-condition-mode teacher
  --local-port-condition-mode predicted
  --local-port-condition-mode mixed
  --local-port-condition-mode both

The default both mode writes teacher and predicted quicklooks and metrics in
one evaluation_summary.json. The predicted metrics are the fair autonomous
forward-model numbers for design inference.


Known simplifications
---------------------
This first implementation is steady only at the training-target level. There
is no transient time, phase, or tau input in the global decoder.

The global model includes a simple baseline internal/interface head for
use_local_surrogate=false. The intended Stage B path is to train Stage A first,
then train Stage B with the local surrogate frozen.

The local/global training templates enable mean/std normalization by default.
Evaluation scripts denormalize predictions before writing plots and summary
errors so visual diagnostics stay in raw physical units.

Stage C joint fine-tuning is only scaffolded by configuration: later runs can
unfreeze the local surrogate and reduce the learning rate after Stage B is
stable.
